### 2.2 碳减排分析模块 (carbon.js)

#### 2.2.1 核心指标汇总接口

- **接口路径**: `GET /carbon/summary`
- **功能描述**: 获取指定周期内的总节电量、总减排量、节能率
- **当前状态**: 后端已实现但不支持周期筛选，始终返回全部数据汇总
- **请求参数**:
  | 参数名 | 类型 | 必填 | 说明 |
  |--------|------|------|------|
  | type | string | 是 | 统计类型：monthly(月度每日)/yearly(年度每月) |
  | period | string | 是 | 统计周期：月度为 "YYYY-MM"，年度为 "YYYY" |

- **返回值结构**:
  ```json
  {
    "code": 200,
    "data": {
      "totalSavedPower": 305.6,
      "totalReducedCO2": 304.6,
      "energySavingRate": 87.8
    }
  }
  ```

- **后端现存问题**:
  1. 不支持 `type` 和 `period` 参数筛选，始终返回全部历史数据汇总
  2. 节电量字段为 `savedEnergy`，减排量字段为 `reducedCo2`，需注意字段名兼容

#### 2.2.2 能耗趋势接口

- **接口路径**: `GET /carbon/trend`
- **功能描述**: 获取趋势数据（月度每日/年度每月）
- **当前状态**: 后端已实现但功能不完整，仅支持返回全部月度汇总数据，缺少周期筛选和日度/年度模式
- **请求参数**:
  | 参数名 | 类型 | 必填 | 说明 |
  |--------|------|------|------|
  | type | string | 是 | 统计类型：monthly(月度每日)/yearly(年度每月) |
  | period | string | 是 | 统计周期：月度为 "YYYY-MM"，年度为 "YYYY" |

- **返回值结构**:
  ```json
  {
    "code": 200,
    "data": [
      {
        "period": "2026-07-01",
        "savedEnergy": 15.2,
        "reducedCo2": 15.1
      }
    ]
  }
  ```

- **详细说明**:
  - **月度每日模式 (type=monthly)**:
    - period 格式：`YYYY-MM`，如 `2026-07`
    - 返回该月内每天的数据，每天一个数据点
    - 每个数据点表示当天的节电量和减排量
    - 数据来源：`carbon_stats` 表的日统计数据（`road IS NULL` 的汇总记录）
    - 数据量：通常为 28-31 条（当月天数）
    - period 字段格式：`YYYY-MM-DD`

  - **年度每月模式 (type=yearly)**:
    - period 格式：`YYYY`，如 `2026`
    - 返回该年内每月的数据，每月一个数据点
    - 每个数据点表示当月的总节电量和总减排量（按月份聚合）
    - 数据来源：`carbon_stats` 表按月份聚合
    - 数据量：最多 12 条
    - period 字段格式：`YYYY-MM`

- **后端现存问题**:
  1. 不支持 `period` 参数筛选，无法指定月份或年份
  2. 不支持 `type=monthly`（日度数据），只返回月度汇总
  3. 不支持 `type=yearly`（年度每月数据）
  4. 返回字段为 `month` 而非 `period`，建议统一为 `period`
  5. CO₂减排量字段为 `co2Reduction`，建议兼容 `reducedCo2`

## 四、前端容错处理说明

对于所有后端缺失的接口，前端已做如下容错处理：

1. **错误静默**: 接口调用失败时不阻塞主流程，使用空数据或默认值兜底
2. **本地缓存**: AI 助手对话历史使用 localStorage 本地持久化

### 5.1 传感器数据 - 近1小时平均值接口

- **接口路径**: `GET /sensor-data/average/{lightId}`
- **功能描述**: 获取指定路灯近1小时内所有传感器采样数据的算术平均值
- **请求参数**:
  | 参数名 | 类型 | 必填 | 说明 |
  |--------|------|------|------|
  | lightId | number | 是 | 路灯ID（路径参数） |
  | startTime | string | 否 | 自定义统计开始时间，格式 `yyyy-MM-dd HH:mm:ss` |
  | endTime | string | 否 | 自定义统计结束时间，格式 `yyyy-MM-dd HH:mm:ss` |

- **数据范围说明**:
  - 默认统计当前时间往前推1小时内的所有采样数据
  - 若传入 `startTime` 和 `endTime`，则按指定区间统计
  - 仅统计该路灯在该时间范围内已有的传感器采样记录

- **返回值结构**:
  ```json
  {
    "code": 200,
    "data": {
      "illuminance": 15234.56,
      "power": 148.32,
      "voltage": 220.15,
      "current": 0.674,
      "temperature": 31.85,
      "humidity": 64.20,
      "samplingEnergy": 1008.6
    }
  }
  ```

- **返回字段说明**:
  | 字段名 | 类型 | 单位 | 说明 |
  |--------|------|------|------|
  | illuminance | number | lux | 平均光照强度 |
  | power | number | W | 平均功率 |
  | voltage | number | V | 平均电压 |
  | current | number | A | 平均电流 |
  | temperature | number | °C | 平均温度 |
  | humidity | number | %RH | 平均湿度 |
  | samplingEnergy | number | Wh | 平均采样间隔耗电量（单位Wh，前端展示时除以1000转换为kWh） |

- **平均值计算逻辑**:
  - 对上述各字段，在指定时间范围内的所有传感器记录取算术平均
  - 若某字段在范围内所有记录均为null，则返回null或省略该字段
  - 若时间范围内无记录，返回空对象或null

- **前端调用位置**: [LightDetail.vue](file:///c:/Users/29674/Desktop/Smart-Street-Light-System-main/Smart-Street-Light-System/frontend/src/views/LightDetail.vue)

### 5.7 跨页面/标签页数据同步（前端加强版实时性方案）

- **功能描述**: 多页面/多标签页打开同一路灯时，任意一处状态变更需即时同步，且详情页保持持续刷新
- **前端实现（三层实时性保障）**:

  **1. BroadcastChannel 跨标签广播（即时推送）**
  - 使用浏览器原生 `BroadcastChannel('smartlight_light_detail')` 跨标签广播
  - 消息格式：`{ type: 'light:updated', lightId: <number>, ts: <number> }`
  - 触发场景：
    - 实时监控页面自动刷新时检测到路灯状态变化
    - 路灯详情页调节亮度成功后立即广播
    - 远程控制等其他页面操作路灯后广播
  - 接收方：
    - 路灯详情页订阅广播，收到对应 `lightId` 时立即调用 `loadAll()` 重新拉取
    - 实时监控页面订阅广播，收到任意 `light:updated` 时自动刷新列表和弹窗

  **2. 详情页定时轮询（持续兜底）**
  - 路灯详情页每 **10 秒** 自动调用 `loadAll()` 刷新全部数据
  - 页面不可见（`document.hidden`）时自动暂停轮询，恢复可见时立即刷新并重启轮询
  - 与 BroadcastChannel 形成互补：广播负责即时通知，轮询负责兜底同步

  **3. 弹窗内数据实时同步**
  - 实时监控页面每次自动刷新后，若设备详情弹窗处于打开状态：
    - 从最新列表数据中查找对应路灯，替换 `selectedLight` 为最新对象
    - 同时重新加载该路灯的传感器数据
  - 弹窗内数据无需手动关闭重开即可实时更新

- **兼容性**: 不支持 BroadcastChannel 的浏览器降级为纯轮询方案（不影响功能）
- **后端建议**: 若需更强实时性（秒级），可结合 WebSocket 推送路灯状态变化事件，前端已预留广播接收逻辑可平滑接入

### 5.9 LocalDateTime 序列化格式问题

- **问题描述**: 后端 `JacksonConfig` 仅配置了 `LocalDate` 和 `LocalTime` 的序列化格式，未配置 `LocalDateTime`，导致返回数组格式 `[2025,7,1,14,0]` 而非标准字符串格式
- **影响范围**: 所有包含 `createTime`、`updateTime` 字段的接口（用户、报警、操作日志等）
- **前端临时修复**: `formatDateTime` 函数兼容数组格式，转换为 `YYYY-MM-DD HH:mm:ss`
- **后端需求**: 在 `JacksonConfig` 中添加 `LocalDateTime` 的序列化配置：
  ```java
  module.addSerializer(LocalDateTime.class, new LocalDateTimeSerializer(DateTimeFormatter.ISO_LOCAL_DATE_TIME));
  module.addDeserializer(LocalDateTime.class, new LocalDateTimeDeserializer(DateTimeFormatter.ISO_LOCAL_DATE_TIME));
  ```
